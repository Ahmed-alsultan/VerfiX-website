VerfiX Integration Guide

This guide provides solutions architects and senior developers with the information needed to design and implement a complete VerfiX integration, including architecture patterns, data flow, security considerations, and operational requirements.

Integration Architecture Patterns

Pattern 1: Synchronous Inline Verification (Standard)

The customer completes identity verification as part of a real-time onboarding flow. The calling application waits for the VerfiX API response before proceeding.

⬡ Diagram: Customer → Onboarding App → VerfiX API (synchronous call) → Verified identity result returned → Onboarding App proceeds with account creation → Core Banking / CRM updated

Suitable for: Most digital onboarding use cases where the customer is actively engaged and the verification can be completed in the session.

Typical processing time: Under 5 seconds for document + face + AML. Set your HTTP client timeout to 30 seconds minimum.

Pattern 2: Asynchronous Verification with Webhooks

The calling application submits the verification request and immediately returns a processing status to the customer. VerfiX sends a webhook notification when verification is complete.

⬡ Diagram: Customer → Onboarding App → VerfiX API (async call) → Application stores verification_id, shows "verification in progress" → VerfiX processes → VerfiX sends webhook to Onboarding App → App updates customer status → Customer notified

Suitable for: High-volume use cases, NFC verification (requires device-side processing time), or cases where processing time may be extended.

Pattern 3: Link-Based Verification

The organisation sends the customer a verification link (generated via the VerfiX API). The customer completes verification from their own device. The organisation receives the result via webhook.

Suitable for: KYC refresh for existing customers, UBO verification during KYB, cases where the customer is not in an active onboarding session.

Security Considerations for Integration

  • ›API key management: Store API keys in environment variables or secret management systems (AWS Secrets Manager, HashiCorp Vault, Azure Key Vault). Never hardcode API keys in source code or commit to version control.
  • ›Image handling: Do not log or store document images or selfie images in your application logs. If you must retain images, apply the same encryption and access controls as VerfiX (AES-256 at rest, TLS in transit).
  • ›Webhook verification: Always verify the X-VerfiX-Signature header on incoming webhook requests before processing. Reject requests with invalid signatures.
  • ›Network: Where possible, whitelist the VerfiX API IP ranges in your outbound firewall rules. Contact VerfiX for the current IP range list.
  • ›Personal data in API calls: Do not include personal data in URL query parameters. Always use POST body (which is encrypted by TLS). Never log raw API responses — they contain personal data.

Webhook Signature Verification

VerfiX signs all webhook payloads with HMAC-SHA256 using your webhook secret. Verify as follows:

Python example:

import hmac, hashlib
def verify_webhook(payload_bytes, signature_header, secret):

expected = hmac.new(secret.encode(), payload_bytes, hashlib.sha256).hexdigest()

return hmac.compare_digest(expected, signature_header)

Production Readiness Checklist

Item Description Status
API key rotation Implement API key rotation procedure. Keys should be rotatable without downtime. Before go-live
Error handling All API error codes handled gracefully. User-facing errors are friendly and informative. Before go-live
Retry logic Exponential backoff implemented for 429 and 500 responses. Before go-live
Webhook processing Webhook endpoint idempotent (handles duplicate delivery). Signature verification implemented. Before go-live
Audit trail storage verification_id and audit_id stored in customer record for every verification. Before go-live
Data Processing Agreement DPA executed with VerfiX before processing real personal data. Before go-live
Image handling Document images and selfies not logged or retained beyond regulatory requirement. Before go-live
Rate limit monitoring X-RateLimit-Remaining monitored. Alerting on approaching limit. Recommended
