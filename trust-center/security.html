Security Overview

Information Security Controls and Architecture

VerfiX Security Overview

This document describes the information security controls, architecture, and practices that VerfiX AG applies to protect customer data, maintain platform integrity, and ensure the confidentiality and availability of the VerfiX identity verification platform.

Security Principles

VerfiX's security programme is built on four principles:

  • Defence in depth: Multiple layers of security controls so that the failure of any single control does not result in a security breach.
  • Least privilege: Every component, user, and process is granted only the minimum access required to perform its function.
  • Data minimisation: Only data necessary for the verification purpose is collected, processed, and retained. Biometric data is not retained beyond the session by default.
  • Auditability: Every significant action on the platform is logged in an immutable, tamper-evident audit trail that can be produced for regulatory inspection.

Encryption

Context Standard Detail
Data at rest AES-256 All customer data and verification records encrypted at rest using AES-256-GCM.
Data in transit TLS 1.3 All API communications use TLS 1.3. TLS 1.2 and below not accepted.
API key storage PBKDF2-SHA256 API keys are hashed using PBKDF2 before storage. Plain-text keys are never stored.
Webhook signatures HMAC-SHA256 All webhook payloads are signed with HMAC-SHA256 for recipient verification.
Key management HSM-backed Encryption keys managed using Hardware Security Modules. Per-deployment key isolation.
Audit trail integrity SHA-256 hash chain Each audit record includes a cryptographic hash that references the previous record, enabling tamper detection.

Data Residency and Infrastructure

All VerfiX processing and storage occurs exclusively within Switzerland:

  • Primary compute: Swiss Tier III certified data centres
  • No cross-border data transfer without explicit customer consent and documented legal basis
  • Infrastructure providers are contractually bound by Swiss data protection law and subject to the same nFADP obligations as VerfiX AG
  • Physical security: Data centre facilities include 24/7 manned security, biometric access controls, and CCTV monitoring
  • Redundancy: Primary and secondary Swiss data centre locations for high availability

Access Control

Control Detail
Role-based access control All VerfiX platform access is governed by RBAC. Roles are defined per function with minimum necessary permissions.
Multi-factor authentication MFA is enforced for all administrative access to production systems. TOTP-based MFA required.
SSO Single Sign-On via SAML 2.0 / OIDC is supported for enterprise customer dashboard access.
IP allowlisting API access can be restricted to specified IP address ranges per API key.
Session management Authenticated sessions are subject to configurable timeout. Session tokens are invalidated on logout.
Privileged access Production system access by VerfiX operations staff requires approval workflow and is logged in the privileged access audit trail.
Vendor access Third-party access to production systems is prohibited. Sub-processor access to customer data is contractually restricted.

Vulnerability Management

  • Penetration testing: Annual third-party penetration testing programme is planned. Results reviewed by security team and remediated per severity SLA.
  • Dependency scanning: Automated scanning of all software dependencies for known vulnerabilities on each code deployment.
  • Static code analysis: Security-focused static analysis integrated into the development pipeline.
  • Responsible disclosure: VerfiX maintains a responsible disclosure programme. Security researchers may report vulnerabilities to security@verfix.ch.
  • Patch management: Critical security patches applied within 24 hours of availability. High severity within 7 days. Medium within 30 days.

Incident Response

VerfiX maintains a documented Incident Response Policy. Key commitments:

Phase VerfiX Commitment
Detection Continuous monitoring of platform health and security signals. Automated alerting on anomalous activity.
Assessment Initial severity assessment within 4 hours of detection.
Containment Containment measures initiated within 8 hours of confirmed incident.
Customer notification Affected customers notified within 72 hours of confirmed breach likely to result in risk to individuals, in accordance with nFADP Art. 24.
Authority notification Notification to FDPIC (Federal Data Protection Commissioner) within the statutory timeframe where required.
Post-incident review Written post-incident report provided to affected customers within 30 days of incident resolution.

Audit Trail

Every significant platform event generates an immutable audit record:

  • Verification initiated — timestamp, API key identifier, IP address, request parameters
  • Verification completed — outcome, all signal values, fraud flags, processing time
  • AML/PEP/sanctions screening — lists checked, match threshold, results, match details where applicable
  • Trust Score generated — score value, component weights, decision outcome, rules triggered
  • Webhook delivery — endpoint, payload hash, delivery status, retry attempts
  • Administrative actions — user ID, action, timestamp, affected resource

Audit records are cryptographically linked (each record includes a SHA-256 hash of the previous record), making any tampering detectable. Records are retained for the period specified in the customer's DPA and can be exported in structured format for FINMA inspection within one hour of request.
