Getting Started with the VerfiX API

VerfiX Developer Documentation

This document provides a practical guide for developers integrating VerfiX identity verification capabilities into applications and systems. It covers authentication, making your first API call, interpreting responses, handling errors, and going from sandbox to production.

Prerequisites

  • ›API credentials: Contact contact@verfix.ch to receive sandbox API credentials. Credentials are issued within 24 hours.
  • ›HTTPS capable: Your integration must make HTTPS requests. HTTP is not accepted.
  • ›JSON: All requests and responses use JSON. Set Content-Type: application/json on all POST requests.
  • ›Base64 encoding: Identity document images and selfie images are submitted as Base64-encoded strings.

Authentication

Include your API key as a Bearer token in every request:

Authorization: Bearer vfx_test_xxxxxxxxxxxxxxxxxxxx  (sandbox)
Authorization: Bearer vfx_live_xxxxxxxxxxxxxxxxxxxx  (production)

API keys have a prefix that identifies the environment. Sandbox keys (vfx_test_) will not function in production and vice versa. If your key is rejected, verify you are using the correct prefix for the environment.

Your First API Call: Document Verification

Step 1: Prepare the document image

Capture an image of the identity document (front side). Acceptable formats: JPEG, PNG, WEBP. Minimum resolution: 640x480px. Recommended: 1080x1440px or higher for best accuracy. Convert to Base64:

Python example:

import base64

with open("id_front.jpg", "rb") as f:

image_b64 = base64.b64encode(f.read()).decode("utf-8")

Step 2: Send the verification request

POST https://sandbox.api.verfix.ch/v1/verify/document
{

"image_front": "<base64_string>",

"country": "CH",

"document_type": "national_id"

}

Step 3: Interpret the response

A successful verification returns HTTP 200 with the following structure:

{

"verification_id": "vfy_3KvB9cH2mNpQrT",

"status": "completed",

"result": "approved",

"document": { "type": "national_id", "country": "CH",

"first_name": "Max", "last_name": "Mustermann",

"date_of_birth": "1985-03-14", "mrz_valid": true },

"fraud_signals": { "fraud_score": 3, "is_tampered": false },

"audit_id": "aud_7mK2pR9xQv"

}

Adding Face Verification and Liveness

After document verification, add face verification to create a FINMA-compliant full biometric check:

POST https://sandbox.api.verfix.ch/v1/verify/face
{

"selfie": "<base64_selfie>",

"reference": "<base64_document_photo_or_selfie_at_enrolment>",

"liveness_check": true

}

The response includes biometric.match_score (0-100) and liveness.is_live (boolean). Store both in your application for the audit trail.

Generating a Trust Score

After document and face verification, generate a composite Trust Score:

POST https://sandbox.api.verfix.ch/v1/trust-score
{

"verification_ids": ["vfy_3KvB9cH2mNpQrT", "vfy_9nXcT4mLqPsR"],

"screening_id": "scr_2KpMxR7vNqBt"

}

Response: trust_score (0-100), decision (APPROVED/REVIEW/REJECTED), and rule_triggers (list of triggered decision rules).

Error Handling

All error responses follow the same structure:

{ "error": { "code": "invalid_request", "message": "image_front is required", "field": "image_front" } }

Always check the HTTP status code first, then the error.code for programmatic handling, and error.message for debugging.

Status Code Recommended handling
400 invalid_request Log error.field and error.message. Fix request before retrying.
401 unauthorized Check API key. Verify correct environment prefix (vfx_test_ vs vfx_live_).
422 unprocessable_image Image quality insufficient. Prompt user to retake with better lighting/focus.
429 rate_limited Implement exponential backoff. Check X-RateLimit-Reset header for retry timing.
500 internal_error Retry with exponential backoff (wait 1s, 2s, 4s). Contact support if persistent.

Going to Production

  • ›Complete integration testing in sandbox. Cover happy path, error cases, and edge cases.
  • ›Request production API credentials: contact@verfix.ch
  • ›Update base URL from sandbox.api.verfix.ch to api.verfix.ch
  • ›Update API key prefix from vfx_test_ to vfx_live_
  • ›Verify webhook endpoints are live and processing events correctly
  • ›Execute Data Processing Agreement with VerfiX before processing real personal data

Support

Developer support is available during integration:

  • ›Email: contact@verfix.ch — include your API key prefix (never the full key) and the request/response that produced the error
  • ›Postman collection: Available on request for rapid API exploration
  • ›Status: status.verfix.ch (planned) for platform availability
