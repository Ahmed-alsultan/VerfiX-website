Platform Architecture and Design Principles

VerfiX Technical Architecture

This document describes the technical architecture of the VerfiX identity verification platform, including its components, data flows, deployment models, and design principles. It is intended for technical evaluators and enterprise architects assessing VerfiX for integration into regulated environments.

Architecture Principles

Principle Description
API-first Every capability is exposed through a documented REST API. No capability exists that is not API-accessible. This ensures full integration flexibility and eliminates vendor lock-in to proprietary interfaces.
Stateless services Each API request is self-contained and does not depend on server-side session state. This enables horizontal scaling and geographic redundancy within Switzerland.
Data minimisation by default The platform is configured by default to process the minimum biometric data necessary for the verification purpose. Raw biometric data (document images, facial photographs) is not retained beyond the session unless explicitly configured.
Immutable audit trail Every significant platform event generates an immutable audit record that cannot be modified or deleted through normal application paths. Records are cryptographically linked to enable tamper detection.
Defence in depth Security controls are applied at multiple layers: network, application, data, and audit. No single control failure results in a security breach.
Swiss residency All compute, storage, and network traffic handling personal data is confined to Swiss territory. No cross-border routing of personal data.

Component Architecture

⬡ Diagram: VerfiX Platform Components — External Layer: Customer Applications (Web / Mobile / API Client) → API Gateway Layer: Load Balancer → API Gateway (Authentication, Rate Limiting, Request Routing) → Service Layer: KYC Service | KYB Service | AML Screening Service | Trust Engine Service | eID Gateway Service → Data Layer: Identity Processing (ephemeral) | Audit Trail Store (immutable) | Configuration Store | Event Queue → Infrastructure Layer: Swiss Data Centre Primary | Swiss Data Centre Secondary (HA)

API Gateway

All external requests pass through the API Gateway layer, which handles:

  • ›TLS termination (TLS 1.3 minimum)
  • ›API key authentication and scoping
  • ›Rate limiting and throttling
  • ›Request logging (metadata only — no personal data in logs)
  • ›Request routing to appropriate service
  • ›Response caching for non-personal-data responses

Service Architecture

Service Function Data handled
KYC Service Document verification, OCR, MRZ extraction, NFC chip reading, face match, liveness detection Document images (ephemeral), biometric data (ephemeral by default), verification results (retained)
KYB Service Company registry lookup, ownership mapping, UBO identification, director verification Company data, UBO identity data, registry responses
AML Screening Service Sanctions, PEP, and adverse media screening against external list providers Name, DOB, nationality (query parameters). No full identity record transmitted to list providers.
Trust Engine Service Risk score aggregation, decision rule application, fraud signal processing Verification signals from other services. No direct personal data — operates on derived scores.
eID Gateway Service EU Digital Identity Wallet credential validation and attribute extraction EU EUDIW credential presentation. Credential validated; attributes extracted and standardised.
Audit Service Immutable event logging, audit trail generation, audit record export Audit metadata, verification outcomes, screening results. Cryptographic linking.

Data Architecture

Ephemeral Processing Layer

Raw biometric data (document images, selfie photographs) is processed in an isolated ephemeral environment. This data is:

  • ›Not written to persistent storage in raw form (by default)
  • ›Processed in memory and immediately discarded after verification is complete
  • ›Never accessible via API after the verification session closes
  • ›Encrypted in transit using TLS 1.3 during transmission to and within the processing layer

Persistent Data Layer

Verification results, risk scores, AML screening results, and audit trail records are stored in the persistent data layer. These records:

  • ›Are encrypted at rest using AES-256-GCM
  • ›Are access-controlled by RBAC — only the owning organisation's API keys can retrieve their own records
  • ›Are retained for the period specified in the customer's DPA (default: 10 years for GwG compliance)
  • ›Are cryptographically linked to enable tamper detection

Deployment Models

Model Architecture Data control Recommended for
Swiss Cloud Multi-tenant managed service in certified Swiss data centres VerfiX manages infrastructure; customer data logically isolated Most organisations — fastest time to value
Private Cloud Single-tenant dedicated infrastructure in Swiss data centres Customer data on dedicated infrastructure; VerfiX manages platform Banks and regulated entities requiring dedicated infrastructure
On-Premise Full platform deployed within customer's own infrastructure Customer controls all infrastructure and data Organisations with the strictest data sovereignty requirements

High Availability and Resilience

  • ›Primary and secondary Swiss data centres in separate geographic locations
  • ›Automatic failover between data centre locations — no manual intervention required
  • ›Stateless service design enables horizontal scaling under load
  • ›API uptime SLA: defined in the service agreement per deployment model
  • ›Audit trail: replicated synchronously to secondary location. No risk of audit record loss.

Network Architecture

  • ›All external communication over TLS 1.3. HTTP not accepted.
  • ›Internal service-to-service communication encrypted and mutually authenticated
  • ›Network segmentation: Processing layer, data layer, and audit layer on separate network segments with firewall rules between segments
  • ›API IP range available for customer firewall allowlisting on request
  • ›DDoS protection applied at the network edge
